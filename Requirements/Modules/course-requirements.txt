1. OVERVIEW & OBJECTIVES
================================================================================

This module delivers a complete Course Management System (CMS) / Learning
Management System (LMS) for the Permaculture Educational Portal. It enables
the organization to create, manage, and deliver courses with a topic-based
learning structure, handle student enrollment through a payment gateway,
manage batches with seat allocation, and conduct live online classes with
recorded video fallback.

KEY OBJECTIVES:
 - Provide a structured, topic-based course delivery platform.
 - Support bilingual content delivery (Bangla & English).
 - Enable a seamless student enrollment-to-payment pipeline.
 - Manage batch-based cohort learning with seat capacity control.
 - Integrate live online classes with YouTube-hosted video recordings.
 - Track student progress at the topic level.
 - Align with ISO 21001:2018 educational management standards.

REFERENCE PLATFORMS:
 - Gaia Education (https://www.gaiaeducation.org)
 - Coursera, Udemy (course structure inspiration)

2. USER ROLES & PERMISSIONS
================================================================================

The system uses the existing Role-Based Access Control (RBAC) system.
The following roles interact with this module:

ROLE: Super Admin
 - Full system access.
 - Can manage all courses, batches, topics, enrollments, payments.
 - Can assign/revoke Course Admin permissions.

ROLE: Course Admin
 - Create, edit, publish, and archive courses.
 - Create and manage batches (set capacity, dates, pricing).
 - Create and manage topics, content, and class schedules.
 - Assign paid students to batches.
 - Upload class recordings and share YouTube links.
 - View enrollment reports, payment reports, and progress analytics.

ROLE: Instructor (Future Enhancement)
 - Assigned to specific batches.
 - Can create/edit topic content for assigned courses.
 - Can conduct and manage live classes.
 - Can view student progress for their batches only.

ROLE: Student (Registered User)
 - Browse available courses (public catalog).
 - Purchase and enroll in courses.
 - Access topic-based learning content for enrolled courses.
 - Join live online classes.
 - Watch recorded class videos (YouTube).
 - Track personal learning progress.
 - Download certificates upon course completion.

ROLE: Guest (Unregistered Visitor)
 - Browse the public course catalog.
 - View course overview, pricing, and batch information.
 - Must register before enrollment or payment.

3. COURSE MANAGEMENT
================================================================================

3.1. COURSE ENTITY FIELDS:
 - Title (JSON: {"en": "...", "bn": "..."})              [Required]
 - Slug (auto-generated from English title)               [Unique]
 - Description / Overview (JSON: {"en": "...", "bn": "..."}) [Required]
 - Short Description (JSON: {"en": "...", "bn": "..."})   [Required]
 - Course Thumbnail Image                                 [Required]
 - Banner Image                                           [Optional]
 - Base Price (BDT) — 0 means FREE                        [Required]
 - Course Level: Foundation | Intermediate | Advanced      [Required]
 - Delivery Mode: Online | Offline | Hybrid                [Required]
 - Duration (e.g., "6 Months", "12 Weeks")                [Required]
 - Total Topics Count (auto-calculated)                   [Read-Only]
 - Total Online Classes Count (auto-calculated)           [Read-Only]
 - Status: Draft | Published | Archived                   [Required]
 - Is Featured (boolean — for homepage highlight)         [Default: false]
 - Created By (foreign key to users)                      [Auto]
 - Meta Title (JSON, for SEO)                             [Optional]
 - Meta Description (JSON, for SEO)                       [Optional]
 - Created At / Updated At                                [Auto]

3.2. COURSE LIFECYCLE:
 Draft → Published → Archived
 - DRAFT: Course is being built. Not visible to students.
 - PUBLISHED: Visible in the public catalog. Students can enroll.
 - ARCHIVED: No new enrollments. Existing students retain access.

3.3. COURSE ADMIN ACTIONS:

 - Edit course details (title, description, price, images, etc.).
 - Change course status (Draft → Published → Archived).
 - Delete course (soft delete — data preserved).
 - Duplicate/clone a course as a template for a new version.
 - View all batches, topics, enrollments, and revenue for the course.

3.4. PUBLIC COURSE CATALOG:
 - List all published courses with thumbnail, title, price, level, duration.
 - Filter by: Level, Delivery Mode, Price (Free/Paid), Status.
 - Search by course title (bilingual search support).
 - Sort by: Newest, Price (Low-High, High-Low), Popularity.
 - Course detail page with full overview, topic list, batch info, and
   enrollment CTA button.

4. BATCH MANAGEMENT
================================================================================

4.1. BATCH ENTITY FIELDS:
 - Batch Title / Name (e.g., "Batch 01 — Spring 2026")   [Required]
 - Course (foreign key)                                   [Required]
 - Start Date                                             [Required]
 - End Date                                               [Required]
 - Enrollment Deadline                                    [Optional]
 - Seat Capacity (0 = Unlimited)                          [Required]
 - Enrolled Count (auto-calculated)                       [Read-Only]
 - Available Seats (capacity - enrolled, auto-calculated)  [Read-Only]
 - Batch Price (BDT) — overrides course base price        [Required]
 - Discount Amount (BDT) or Discount Percentage           [Optional]
 - Status: Upcoming | Ongoing | Completed | Cancelled     [Required]
 - Is Enrollment Open (boolean)                           [Default: true]
 - Class Schedule / Routine (see Section 6)               [Optional]
 - Created At / Updated At                                [Auto]

4.2. BATCH LIFECYCLE:
 Upcoming → Ongoing → Completed
 - UPCOMING: Enrollment is open. Classes have not started.
 - ONGOING: Classes are in progress. Enrollment may still be open if seats
   are available.
 - COMPLETED: All classes and topics are done. No new enrollments.
 - CANCELLED: Batch was cancelled. Refund process applies.

4.3. BATCH ADMIN ACTIONS:
 - Create batch under a specific course.
 - Set seat capacity and enrollment deadline.
 - Set batch-specific pricing (override course base price).
 - Manually assign paid students to the batch.
 - View enrolled students list with payment status.
 - Change batch status.
 - Close/open enrollment manually.
 - Export student list (CSV, Excel, PDF).

4.4. SEAT ALLOCATION & ENROLLMENT LOGIC:
 - When a student pays → status = "paid, awaiting assignment".
 - Course Admin manually reviews payment → assigns student to a batch.
 - System checks: if (batch.enrolled_count < batch.capacity) → allow.
 - If batch is full → Admin cannot assign more students. Must open a
   new batch or increase capacity.
 - Students can only be assigned to ONE batch per course.

5. TOPIC & CURRICULUM STRUCTURE
================================================================================

5.1. TOPIC ENTITY FIELDS:
 - Title (JSON: {"en": "...", "bn": "..."})               [Required]
 - Course (foreign key)                                   [Required]
 - Description / Summary (JSON: {"en": "...", "bn": "..."}) [Optional]
 - Order Index (integer — for sequential display)         [Required]
 - Topic Type: Content | PDF | Online Class | Assignment | Quiz [Required]
 - Content Body (JSON: {"en": "...", "bn": "..."})        [Optional]
   (Rich HTML — text, tables, images, embedded video/audio)
 - Attachments / Resources (downloadable files)           [Optional]
 - Estimated Duration (e.g., "45 minutes", "2 hours")     [Optional]
 - Is Published (boolean)                                 [Default: false]
 - Is Free Preview (boolean — accessible without enrollment) [Default: false]
 - Created At / Updated At                                [Auto]

5.2. TOPIC TYPES EXPLAINED:
 TYPE: Content
   - Standard learning material (text, images, tables, video, audio).
   - Students read/watch at their own pace.

 TYPE: PDF (Document Reader)
   - Admin uploads a PDF file as the topic's primary learning material.
   - Students read the PDF via an embedded in-browser PDF reader.
   - PDF is displayed inline within the topic content page (no download
     required, though a download option is also available).
   - Supports bilingual PDFs — admin can upload separate EN and BN files.
   - Use case: Course handouts, reference guides, textbook chapters,
     research papers, or supplementary reading material.

 TYPE: Online Class (Live Session)
   - Linked to a scheduled live class (see Section 6).
   - Displays class date/time, join link, and video recording after class.

 TYPE: Assignment (Future Enhancement)
   - Submission-based task for students.
   - Can include file upload and text response.

 TYPE: Quiz (Future Enhancement)
   - Auto-graded assessment.
   - Multiple choice, true/false, short answer.

5.3. CURRICULUM DISPLAY (STUDENT VIEW):
 - On the Student Dashboard → "My Courses" section.
 - Each enrolled course shows a topic list (ordered by order_index).
 - Example: A course with 10 topics → student sees 10 items in a list.
 - Each topic shows:
   * Title (in student's preferred language)
   * Type icon (Content / PDF / Online Class / Assignment / Quiz)
   * Duration estimate
   * Completion status (Not Started / In Progress / Completed)
   * Lock icon if the topic is not yet published or requires a prerequisite.
 - Clicking a topic → opens the full topic content page.

5.4. TOPIC ORDERING:
 - Topics are displayed in ascending order_index.
 - Admin can reorder topics via the drag-and-drop interface.
 - Sequential access (optional): Admin can enforce that students must
   complete Topic N before accessing Topic N+1.

6. LIVE ONLINE CLASS MANAGEMENT
================================================================================

6.1. CLASS SCHEDULE / ROUTINE:
 - Each batch can have a class routine (weekly schedule).
 - Class Routine Fields:
   * Day of Week (Saturday, Sunday, Monday, etc.)
   * Start Time (e.g., 08:00 PM)
   * End Time (e.g., 10:00 PM)
   * Timezone (default: Asia/Dhaka, BST UTC+6)
   * Platform: Zoom | Google Meet | Microsoft Teams | Other
   * Recurring (boolean — weekly recurring)
 - Admin sets the routine when creating/editing the batch.

6.2. INDIVIDUAL CLASS SESSION ENTITY:
 - Title (e.g., "Topic 3 — Soil Regeneration Live Class") [Required]
 - Batch (foreign key)                                    [Required]
 - Topic (foreign key — linked to a specific topic)       [Required]
 - Scheduled Date & Time                                  [Required]
 - Duration (minutes)                                     [Required]
 - Meeting Link (Zoom/Meet/Teams URL)                     [Required]
 - Meeting Password (optional, encrypted)                 [Optional]
 - Status: Scheduled | Live | Completed | Cancelled       [Required]
 - YouTube Video URL (added after class is completed)     [Optional]
 - Video Upload Date                                      [Optional]
 - Instructor Name                                        [Optional]
 - Notes / Agenda (JSON: {"en": "...", "bn": "..."})      [Optional]
 - Attendance (many-to-many with students)                [Auto]
 - Created At / Updated At                                [Auto]

6.3. CLASS WORKFLOW:
 BEFORE CLASS:
   - Admin creates class sessions linked to topics in a batch.
   - System auto-generates sessions from the batch routine (optional).
   - Students see upcoming classes on their dashboard with countdown timer.
   - System sends reminder notification (email/SMS) before class (e.g., 1hr).

 DURING CLASS:
   - Students click the "Join Class" button to open the meeting link.
   - Attendance is optionally tracked (join = mark present).

 AFTER CLASS:
   - Admin marks class as "Completed".
   - Admin uploads the class recording to YouTube (unlisted/private).
   - Admin adds the YouTube video URL to the class session record.
   - The video link becomes visible on the topic content page.
   - Students who missed the class can watch the recorded video.

6.4. RECORDED CLASS VIDEO ACCESS:
 - Videos are hosted on YouTube (unlisted for privacy).
 - Only enrolled students of the specific batch can see the video link.
 - Video link appears under the corresponding topic in the curriculum.
 - Students can watch anytime after the video is uploaded.

7.3. CONTENT VERSIONING (Future Enhancement):

8. STUDENT ENROLLMENT & REGISTRATION FLOW
================================================================================

8.1. ENROLLMENT PIPELINE:

 SCENARIO A — REGISTERED & LOGGED-IN STUDENT + PAID COURSE:
   1. Student browses the course catalog.
   2. Student clicks "Enroll Now" on a course detail page.
   3. System shows available batches with pricing, dates, and seats.
   4. Student selects a batch.
   5. System redirects to Payment Page.
   6. Student completes payment via payment gateway.
   7. Payment status updated to "Completed".
   8. System creates an enrollment record with status "Paid — Awaiting
      Batch Assignment".
   9. Course Admin receives notification of new paid enrollment.
   10. Course Admin reviews and assigns the student to the selected batch.
   11. Student receives confirmation notification.
   12. The student can now access course content and classes.

 SCENARIO B — REGISTERED & LOGGED-IN STUDENT + FREE COURSE:
   1. Student clicks "Enroll Now" on a free course.
   2. System shows available batches.
   3. Student selects a batch.
   4. System auto-enrolls the student (no payment required).
   5. The student can immediately access course content.

 SCENARIO C — GUEST (UNREGISTERED) VISITOR + ANY COURSE:
   1. Guest browses the course catalog.
   2. Guest clicks "Enroll Now".
   3. System redirects to Registration Page (with course context preserved).
   4. Guest completes registration (name, email, phone, password).
   5. Email verification sent (if enabled).
   6. After registration → system redirects back to the enrollment flow
      (Scenario A or B depending on course pricing).
   7. The entire flow feels seamless — register → pay → enroll.

8.2. ENROLLMENT ENTITY FIELDS:
 - User (foreign key)                                     [Required]
 - Batch (foreign key)                                    [Required]
 - Payment (foreign key, nullable for free courses)       [Optional]
 - Enrollment Status:                                     [Required]
     * pending_payment — awaiting payment completion
     * paid_awaiting_assignment — paid but not yet assigned to batch
     * active — assigned to batch and can access content
     * completed — finished all topics
     * dropped — student dropped out
     * suspended — temporarily blocked by admin
     * refunded — payment refunded and access revoked
 - Enrolled At (timestamp)                                [Auto]
 - Completed At (timestamp, nullable)                     [Optional]
 - Last Accessed At (timestamp)                           [Auto]
 - Progress Percentage (0-100, auto-calculated)           [Read-Only]
 - Created At / Updated At                                [Auto]

8.3. ENROLLMENT CONSTRAINTS:
 - A student can enroll in a course only ONCE (one batch per course).
 - Unique constraint: (user_id, batch_id) composite unique.
 - System must check seat availability before confirming enrollment.
 - Admin can manually change enrollment status (e.g., suspend, drop).

9. PAYMENT & BILLING SYSTEM
================================================================================

9.1. PAYMENT ENTITY FIELDS:
 - User (foreign key)                                     [Required]
 - Batch (foreign key)                                    [Required]
 - Amount (BDT, decimal 10,2)                             [Required]
 - Discount Applied (BDT)                                 [Optional]
 - Net Amount (amount - discount)                         [Read-Only]
 - Payment Gateway: bKash | Nagad | Rocket | SSLCommerz   [Required]
   | Bank Transfer | Cash | Manual
 - Transaction ID (from gateway)                          [Optional]
 - Gateway Response (JSON — raw gateway response data)    [Optional]
 - Payment Status:                                        [Required]
     * pending — payment initiated but not confirmed
     * completed — payment verified and confirmed
     * failed — payment attempt failed
     * refunded — full refund processed
     * partially_refunded — partial refund processed
 - Payment Date (timestamp)                               [Auto]
 - Verified By (admin user_id, for manual verification)   [Optional]
 - Verification Notes                                     [Optional]
 - Invoice Number (auto-generated, e.g., INV-2026-00001)  [Auto]
 - Created At / Updated At                                [Auto]

9.2. PAYMENT FLOW:
 1. Student selects a batch → system calculates price (batch price or
    course base price, minus any applicable discount).
 2. System creates a payment record with status "pending".
 3. Student is redirected to the selected payment gateway.
 4. On successful payment → gateway sends a callback/webhook.
 5. System verifies the transaction and updates the status to "completed".
 6. System updates enrollment status to "paid_awaiting_assignment".
 7. Admin receives notification.

 MANUAL PAYMENT (Bank Transfer / Cash):
 1. Student selects "Bank Transfer" or "Cash" as payment method.
 2. System shows bank account details or payment instructions.
 3. The student makes the payment offline.
 4. Student submits transaction reference/receipt upload.
 5. Admin manually verifies and marks the payment as "completed".

9.3. REFUND POLICY:
 - Admin can initiate full or partial refunds.
 - Refund updates payment status and revokes enrollment access.
 - Refund history is maintained for audit purposes.

9.4. REVENUE REPORTING (Admin):
 - Total revenue by course, batch, time period.
 - Payment status breakdown (completed, pending, failed, refunded).
 - Export reports to CSV, Excel, PDF.

11. PROGRESS TRACKING & ANALYTICS
================================================================================

11.1. STUDENT PROGRESS (TOPIC-LEVEL):
 - Track per student, per topic: Not Started | In Progress | Completed.
 - "topic_progress" table:
   * User ID (foreign key)
   * Topic ID (foreign key)
   * Status: not_started | in_progress | completed
   * Started At (timestamp, nullable)
   * Completed At (timestamp, nullable)
   * Time Spent (seconds, cumulative)
   * Unique constraint: (user_id, topic_id)

11.2. COURSE-LEVEL PROGRESS:
 - Auto-calculated: (completed_topics / total_topics) × 100 = percentage.
 - Updated whenever a topic is marked as completed.
 - Stored on the enrollment record for quick dashboard display.

11.3. ADMIN ANALYTICS DASHBOARD:
 - Total enrollments (by course, by batch, by time period).
 - Completion rates (percentage of students who complete the course).
 - Average progress per batch.
 - Revenue statistics.
 - Most popular courses (by enrollment count).
 - Student engagement metrics (last access, time spent).
 - Dropout rate analysis.
 - Export analytics reports (CSV, Excel, PDF).

11.4. STUDENT ANALYTICS (Self-View):
 - Personal progress across all enrolled courses.
 - Completion percentage per course.
 - Topics completed vs. remaining.
 - Class attendance record.
 - Time spent on learning.

13. BILINGUAL (i18n) REQUIREMENTS
================================================================================

13.1. CONTENT LANGUAGE SUPPORT:
 - All user-facing content must be stored in BOTH English and Bangla.
 - JSON field format: {"en": "English text", "bn": "বাংলা টেক্সট"}
 - Fields requiring bilingual storage:
   * Course: title, description, short_description, meta_title, meta_desc
   * Topic: title, description, content_body
   * Class Session: notes/agenda
   * Notification templates

13.2. LANGUAGE SWITCHING:
 - Site supports language toggle (EN ↔ BN) via the existing locale system.
 - Students' preferred language is stored in their profile.
 - Content automatically displays in the user's selected language.
 - Fallback: If Bangla content is empty → display the English version.

13.3. ADMIN CONTENT ENTRY:
 - Admin content forms must show side-by-side or tabbed input fields for
   both languages.
 - Bangla is mandatory. English is strongly recommended but not mandatory.
 - Rich content editor must support Bangla Unicode input (already supported
   by modern browsers).

13.4. EXISTING INFRASTRUCTURE:
 - Translation file: lang/bn.json (for UI labels, buttons, nav items).
 - Translation file: lang/en/ (default Laravel English messages).
 - Locale middleware already configured.
 - The HasTranslations trait is already implemented on the Course model.
